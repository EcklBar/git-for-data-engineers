Introduction to Git

What is Version Control?

  • Stores and manages every revision of files and code
  • Collaborate on source code & manage releases
  • Roll back to previous versions when bugs are discovered
  • VSC = Version Control System

About Git

  • Created by Linus Torvalds in 2005 (for the Linux Kernel)
  • Replaces older VCS tooling (CVS, Subversion, BitKeeper, Mercurial)
  • Focuses on source code distribution (everyone has a copy)
  • Implements efficient file storage
  • Ensures files integrity

Git Advantages

  • Distributed development (work on your local copy)
  • High performance
  • Ease of use
  • Strong support for branching and history management
  • Extensive tooling for advanced operations

Git Challenges

  • A new code management model
  • Learning curve (mind shift & commands)
  • Destructive operations when used incorrectly
  • Limited support for binary files
🚀 VCS Services

GitHub

  • Web-based hosting service for version control using Git
  • Access control, issue tracking, documentation, automation
  • Leading VCS service for open-source and enterprises
  • Acquired by Microsoft in 2018

Alternatives

  • GitLab
  • BitBucket

Git Basics

🔄 The Distributed Model
Clone Repository
→
Make Local Changes
→
Commit & Push
  • Get a copy of the entire repository from the server ("clone")
  • All the content and all the history
  • Make changes locally (commits and branches)
  • Push changes back to the server, a.k.a. remote repository

Git Object Types

  • Blob: The raw content of a file – without file metadata
  • Tree: One level of directory information (file names, blobs, subdirectories)
  • Commit: The version of the tree + metadata (author, dates, log messages, etc.)
  • Tag: A label associated with a commit, e.g., a version number

Delta Storage

  • Each file is stored as a complete, separate object when introduced
  • Changes to an existing file are stored as deltas
  • Git uses a smart and efficient mechanism for storing changes
  • Results in significant space savings and efficient history management
git init
- Initialize a new Git repository
git clone <repository_url>
- Copy an existing Git repository
git add <file>
- Stage files (add to the index)
git commit -m "Commit message"
- Record staged changes
git status
- Check the status of your working directory

Git Workflow and Three Stages

📁 The Three Stages
Working Directory
→
Staging Area (Index)
→
Repository

Working Directory

  • Create, edit, and delete files in your project
  • Think of it as your "sandbox" where you work on your code and data

Staging Area (Index)

  • A checklist of changes you want to commit
  • Prepare and select specific changes to be included in the next commit
  • Using git add, you move changes from working directory to staging area

Repository

  • The repository stores your project's complete history
  • When you git commit, the changes in the staging area are recorded as a new snapshot in the repository
  • Your project's complete history is stored here

File States

  • Untracked: A new file that is first introduced
  • Unmodified: An existing file with no local changes
  • Modified: An existing file with local changes
  • Staged: Files with local changes marked for the next commit
  • Committed: Changes safely stored in the local database

Git Branches

🌳 Branching Strategy
  • Treat the source code as a tree of branches
  • Each branch is a linear set of commits
  • Use a side branch to work on a private version of the code
  • Merge the side branch back into the main branch
  • Collaborate with other developers
git checkout branch_name
- Switch to branch
git checkout -B branch_name
- Create and switch to new branch
git branch -D testing
- Delete branch
git log --oneline --decorate --graph --all
- View all branches
git show branch_name
- Show branch head

Merge Branches

  • Once ready, a side branch is merged into the main branch
  • Switch to the main branch: git checkout main
  • Merge the side branch: git merge side_branch

Divergent History

  • Collaborating using multiple branches may lead to divergent history
  • Developer #1 starts a new branch and changes a file
  • Developer #2 adds a new commit to the main branch and also changes the same file
  • Developer #1 cannot merge because the changes conflict

Merge Conflicts

⚡ Facing Conflicts
  • When a merge conflict emerges, the merge is paused to let you resolve it
  • Use git status to see which files have conflicts
  • Files without conflicts are staged
  • Files with conflicts appear as "Both modified"
Conflict Example:
<<<<<<< HEAD
new text on main
=======
text from side branch
>>>>>>> my_side_branch

Resolving Conflicts Manually

  • Open the file that has conflicts
  • Delete/Change irrelevant lines
  • Stage the file (git add)
  • To finish the merge when all conflicts are resolved: git merge --continue

Resolving With Style

  • Modern IDE viewing and resolving merge conflicts (e.g., VSCode)
  • Using a merge editor is the better practice
  • See the changes more clearly
  • Easily choose the preferred version
  • Combine code from both branches
  • More intuitive and human-friendly interface
🔧 VSCode Merge Example
  1. Open the file in conflict
  2. Press "Resolve in Merge Editor"
  3. Compare the changes and resolve the conflict
  4. Choose "Accept Incoming" to select changes from the side branch
  5. Choose "Accept Current" to select changes from the main branch
  6. Manually update the result if needed
  7. Once ready, press "Complete Merge" to stage the file
  8. Run git merge --continue or use VSCode to finish

Essential Git Commands

Repository Management

git init
- Create a new local repository
git clone
- Download a copy of a remote repository
git status
- Show the status (index) of the working directory

Staging & Committing

git add
- Stage files (add to the index)
git reset
- Unstage files (remove from the index)
git commit
- Create a new commit (changeset) in your local repository

Branching

git checkout
- Set your working directory to work on a different commit, tag or branch
git log
- View the history of your local repository (commits)
git diff
- Compare versions (local changes, staged changes, different commits, etc.)

Remote Operations

git fetch
- Bring info (commits, branches) from the remote repository
git pull
- Fetch and checkout the latest commit of your current branch
